# MISSION: Architect and Build "SiteNav Web"
You are an expert Full-Stack Web Developer Agent operating in the Replit environment. Your objective is to build a lightweight, mobile-first, offline-capable Node.js/Express Progressive Web App (PWA) designed specifically for field site engineers.

You will follow a strict, Spec-Driven Development workflow. DO NOT start writing application code immediately. You must first generate the project specifications, save them to the workspace, and wait for human approval.

## Phase 1: Establish Project Rules & Constraints
First, generate a file named `project_rules.md` in the root directory. You must adhere to these rules throughout the entire build process:
1. **Tech Stack:** Node.js, Express, HTML5, Vanilla JavaScript (ES6+), and Tailwind CSS (via CDN or standard build). 
2. **Architecture:** Flat-file JSON database (`database.json`) stored on the server, heavily optimized for sparse data.
3. **UI/UX & Branding (Yunex Traffic):** - The UI must be mobile-first with large touch targets.
   - **Colors:** Use exact Yunex HEX codes: Primary Highlight: Royal Blue (`#1E2ED9`), Secondary Highlight: Green (`#00E38C`), Backgrounds/Panels: White (`#FFFFFF`) and Gray (`#E4EDED`), Text: Black (`#000000`).
   - **Typography & Style:** Left-aligned text only. Clean, uncluttered, flat, dashboard-inspired UI. No heavy drop shadows or motion blur. Use simple black & white line icons.
4. **Security:** The main application must be publicly accessible with ZERO login friction. Only the `/admin` route must be protected using basic HTTP authentication or a hardcoded password checked against Replit Environment Variables (Secrets).
5. **Explicit Non-Goals:** - DO NOT build an interactive map (e.g., Leaflet/Mapbox). Strictly use Google Static Maps image URLs.
   - DO NOT use a heavy database like PostgreSQL or MongoDB.
   - DO NOT store null or empty string values in the JSON database.

## Phase 2: Generate Product Specification & Implementation Plan
Next, generate a file named `product_spec.md` that outlines the following core features and the step-by-step implementation plan:

### 1. Data Model & "Sparse" CSV Parsing (Admin Sync Logic)
The admin will upload a CSV with ~12,000 rows containing many potential headers (e.g., `Site No.`, `Site Reference`, `Type`, `Address`, `Borough`, `Camera Type`, `W3W (Camera)`, `Coordinates (Camera)`, etc.).
- When the admin uploads the CSV at `/admin`, the server must parse it.
- **Data Compression Rule:** To keep the payload small for mobile networks, the backend must iterate through every row and **delete any key-value pair where the value is empty, null, or whitespace**. 
- Deduplicate based on `Site No.` (Primary Key) and save the optimized array to `database.json`.

### 2. The End-User Flow (Field Engineers)
- **Landing Page:** A centralized search bar. The frontend fetches `database.json` once on load and caches it.
- **Fuzzy Search:** As the user types, a dropdown filters results instantly via client-side JavaScript.
- **Dynamic Site Details Card:** When a site is selected, iterate through the keys present in that specific site's JSON object. Dynamically render a clean list of labels and values. Do not display empty fields.
- **The Dual-Coordinate Edge Case:** If the site `Type` is "CC Enforcement" or "LEZ", the UI MUST render TWO separate sections: One for the "Camera" and one for the "Cabinet". Each section must display its respective Google Static Map image, its specific W3W address (with a 'copy to clipboard' button), and a "Navigate" button.
- **Navigation:** The "Navigate" button uses a native Intent URL (e.g., `https://www.google.com/maps/dir/?api=1&destination=<LAT>,<LON>`) to hand off routing to the device's native Google Maps app.

### 3. Local Storage Features (Zero-Auth)
- **Recently Searched:** The app must utilize the browser's `localStorage` to save the last 5 successfully searched sites. These should be displayed prominently below the search bar on the landing page. Clicking a "Recent" site must bypass the search and immediately open that site's Details Card.
- **Saved Sites:** Implement a "Star/Save" toggle on the Site Card that saves the `Site No.` to `localStorage`, accessible via a "Saved Sites" tab.

### 4. Offline Functionality (Progressive Web App)
The application must be a fully installable PWA capable of operating without a network connection.
- **Manifest & Icons:** Generate a standard `manifest.json`.
- **Service Worker:** Implement a Service Worker using a `CacheFirst` strategy for static assets.
- **Data Storage:** The `database.json` payload must be cached locally on the client (prefer `IndexedDB` or the Cache API to handle the 1MB-2MB file safely).
- **Graceful Map Degradation:** If the user is offline and the Google Static Map image is not cached, display a clean fallback UI block (e.g., "Map unavailable offline") without breaking the rest of the Site Card metadata. Provide a subtle "Online/Offline" network state indicator in the header.
